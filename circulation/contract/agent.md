Natural Language Contracts
====

Natural language contracts are SatoshiNet smart contracts supported by AI Agents. Contract content is written as natural language agreements. AI Agents perform structured interpretation, verification, and execution-conclusion generation.

Natural language contracts follow the common [Smart Contracts](readme.md) protocol. This document defines their boundary and first-phase requirements.


Naming
----

Recommended names:

1. Chinese name: 自然语言合约.
2. English name: Natural Language Contract.
3. Technical shorthand: Agent contract.
4. Code type: `ContractTypeAgent`.

Protocol documents use "Natural Language Contract". Code types and internal implementation may use `Agent`.


Contract Type
----

Natural language contract type:

```text
ContractTypeAgent
```

Natural language contracts are not EVM contracts. Users do not need to write Solidity, ABI, or bytecode.

Natural language contracts are not template contracts. Contract logic does not need to be hard-coded in SatoshiNet node source code in advance.

Natural language contracts are not channel contracts. They do not depend on RSMC channels, multi-party signatures, or commitment transactions.


CoreNode Agent
----

SatoshiNet natural language contracts are supported by a protocol built-in AI Agent. In the first phase, the whole SatoshiNet network can be understood as having one protocol Agent, executed by the built-in CoreNode.

The CoreNode identity is determined by the CoreNode public key built into the protocol implementation. In the current reference implementation, the CoreNode public key corresponds to `GetCoreNodePubKey()` in `indexer/common/const.go`.

Prediction confirmations generated by CoreNode must carry the CoreNode public key and CoreNode signature. When validating `confirm`, nodes must check that the submitted public key equals the protocol-configured CoreNode public key, that the Taproot address derived from that public key equals the protocol-configured CoreNode address, and that the signature covers the current contract address and confirmation parameters. The signature does not replace canonical Result verification, but proves that the Agent conclusion was produced by the protocol-recognized CoreNode.

CoreNode has the following roles in natural language contracts:

1. Agent executor.
2. Reviewer of whether contract content is understandable, verifiable, and executable.
3. Protocol oracle.
4. Authorized entity for generating natural language contract execution conclusions.

Ordinary users cannot replace CoreNode to generate protocol-recognized Agent conclusions. Ordinary users may deploy contracts, submit materials, request execution, or raise disputes, but cannot call built-in Agent interfaces that are restricted to CoreNode.

The protocol may later be extended to multi-Agent verification, for example accepting an execution result when at least 2 of 3 Agents give the same conclusion. The testing phase may use a single CoreNode Agent first.


Contract Content
----

Natural language contract content is an executable agreement text. The text must be parseable by AI Agents into a verifiable, acceptable, and executable structure.

The agreement should contain:

1. contract purpose.
2. participants and roles.
3. target assets.
4. how assets enter the contract.
5. trigger conditions.
6. verification data sources.
7. acceptance criteria.
8. execution method.
9. timeout, failure, and dispute handling.
10. possible asset-transfer results.

The contract may include structured helper fields:

1. asset list.
2. participant address list.
3. time or block-height limits.
4. allowed data sources.
5. verifier or arbitrator rules.
6. contract version.

The natural language text is the primary source of contract semantics. Structured fields help parsing but must not change the core meaning of the text.


Deployment
----

`CONTRACT_DEPLOY` deploys a natural language contract.

Deploy payload contains:

1. contract type.
2. agreement text.
3. contract version.
4. deployer.
5. deploy random value.
6. optional structured helper fields.

A natural language contract does not become executable automatically after deployment. A successful deployment only creates a pending contract. Its initial state is `PendingReady`.

All natural language contracts must provide a fixed CoreNode activation interface:

```text
ready
```

The `ready` interface can only be called by CoreNode. Other addresses are not authorized to call it. Invoker identity is still resolved from the last input of the call transaction according to the common rule.

When CoreNode calls `ready`, it must check:

1. whether content is complete.
2. whether conditions are verifiable.
3. whether acceptance criteria are explicit.
4. whether execution results can be mapped to Asset Intents.
5. whether the contract depends on private facts that cannot be verified or agreed by consensus.
6. whether it contains contradictory clauses, circular conditions, or non-executable clauses.

If the check passes, CoreNode interaction generates a canonical `CONTRACT_RESULT`. The Result declares that the contract enters `Ready` state. Only natural language contracts in `Ready` state can accept ordinary business calls.

If the check fails, CoreNode interaction generates a rejection result, and the contract enters `Rejected` or `Invalid` state. A contract that has not entered `Ready` state must not produce asset-transfer execution results.


Invocation
----

`CONTRACT_INVOKE` submits events, proofs, acceptance requests, execution requests, or dispute materials to a natural language contract.

Invoke content may contain:

1. trigger action.
2. invoker identity.
3. on-chain proof.
4. external proof.
5. acceptance material.
6. execution request.
7. dispute or appeal material.

The call transaction must output to the invoked contract address as gas/funding output. Invoker identity is resolved from the last input of the call transaction.

Natural language contract calls are divided into two categories:

1. CoreNode built-in calls: including `ready` and future Agent-only interfaces defined by the protocol. Only CoreNode can call them.
2. Ordinary business calls: submitted by participants or other authorized addresses to provide events, proofs, acceptance materials, execution requests, or dispute materials.

Except for `ready`, ordinary business calls only take effect after the contract is in `Ready` state.


AI Agent Execution
----

AI Agents generate structured execution conclusions from agreement text, contract state, trigger conditions, proofs, and on-chain environment.

In the first phase, the Agent is executed by CoreNode. Agent conclusions generated by CoreNode enter the SatoshiNet contract execution flow and are expressed through canonical `CONTRACT_RESULT`. Agent output cannot be free text only; it must be structured, verifiable, comparable, and replayable.

Execution conclusions must contain:

1. whether trigger conditions are satisfied.
2. verification materials used.
3. whether acceptance passed.
4. Asset Intents to generate.
5. whether more materials are required.
6. whether the contract enters failure, timeout, or dispute state.
7. conclusion summary.

AI Agents cannot directly spend UTXOs. Asset transfers must be settled by canonical `CONTRACT_RESULT`.

The concrete Agent execution method still needs to be defined, including at least:

1. model version or Agent implementation version.
2. fixed prompts and interpretation rules.
3. structured output schema.
4. output signature or CoreNode identity-verification method.
5. switching rules between single-Agent mode and multi-Agent threshold mode.
6. failure, timeout, dispute, and retry rules.


Verifiability
----

Natural language contracts must avoid facts that cannot reach consensus.

Allowed data sources include:

1. SatoshiNet on-chain transactions.
2. contract state.
3. specified on-chain contracts or oracle results.
4. external proofs specified at deployment.
5. multi-party signed acceptance materials.
6. AI Agent verification results allowed by protocol.
7. verification results provided by CoreNode as protocol oracle.

If multiple nodes may reach different conclusions from the same material, the protocol must provide constraints:

1. fixed model version.
2. fixed prompts and interpretation rules.
3. fixed structured output format.
4. multi-Agent consensus.
5. arbitrator or challenge period.
6. only independently verifiable proofs are accepted.


Result TX and State
----

Natural language contracts do not accept externally submitted `CONTRACT_RESULT` transactions in mempool.

The block producer constructs canonical `CONTRACT_RESULT` from AI Agent execution conclusions. Validating nodes must verify the conclusion according to protocol-defined Agent verification rules and compare Result TX.

Natural language contract results enter the SatoshiNet contract-result flow. Whether the result is a `ready` activation result, rejection result, ordinary execution result, failure result, or dispute result, it must be expressed through canonical `CONTRACT_RESULT`. External users cannot directly submit non-canonical Result TXs.

Natural language contract state contains at least:

1. agreement text hash.
2. contract version.
3. current execution state.
4. submitted proofs.
5. accepted items.
6. pending items.
7. generated results.
8. dispute or failure state.

Natural language contract states should use a unified status definition as much as possible. Different contracts may use only a subset of the states, but the same state name must have the same meaning across natural language contracts.

Base states include:

1. `PendingReady`: the contract has been deployed and waits for CoreNode to call `ready`.
2. `Ready`: CoreNode has confirmed that the contract content is understandable, verifiable, and executable. Ordinary business calls are accepted.
3. `Rejected`: CoreNode rejected activation. The contract cannot execute ordinary business calls.
4. `Invalid`: contract content, structure, or invocation material is invalid and cannot enter later execution.
5. `PendingEvidence`: the contract waits for participants to submit proofs or acceptance materials.
6. `PendingExecution`: trigger conditions are satisfied, waiting for Agent execution conclusion or Result settlement.
7. `Completed`: main obligations are completed and related asset transfers have been settled.
8. `Failed`: the contract ends due to failure conditions, timeout, or non-executable reasons.
9. `Disputed`: the contract enters dispute state and waits for CoreNode, arbitrator, or protocol-defined dispute processing.
10. `Expired`: the contract has exceeded its validity period and no continuation condition is satisfied.

Agent contract state roots must be included in the unified contract state root.


Prediction Natural Language Contracts
----

The first phase prioritizes prediction natural language contracts. A prediction natural language contract is a fixed subtype under `ContractTypeAgent`:

```text
prediction
```

A prediction natural language contract expresses multiple candidate outcomes for a verifiable event. Users bet by transferring assets to the contract address. After the event ends, the Agent confirms the final result based on the data source specified at deployment. The contract then automatically distributes the prize pool or refunds users.

The Agent of a prediction natural language contract is only responsible for:

1. `ready`: reviewing whether the contract content can be understood, verified, and executed.
2. `confirm`: submitting the final result based on the specified data source.

Bet records, fees, prize distribution, refunds, and contract closure are deterministically handled by the prediction contract runtime.

### Deploy Payload

Prediction natural language contract deploy payload contains:

```json
{
  "subtype": "prediction",
  "title": "...",
  "description": "...",
  "time_base": "unix",
  "event_time": 1780310400,
  "bet_deadline": 1780306800,
  "confirm_after": 1780396800,
  "source_url": "https://xxx.com",
  "bet_asset": "::",
  "min_bet_unit": "10000",
  "outcomes": [
    {"id": "a", "text": "..."},
    {"id": "b", "text": "..."},
    {"id": "c", "text": "..."}
  ]
}
```

Field rules:

1. `subtype` must be `prediction`.
2. `title` and `description` describe the prediction target.
3. `time_base` specifies the time base. It can be `unix` or `height`.
4. `event_time`, `bet_deadline`, and `confirm_after` are interpreted according to `time_base`.
5. `source_url` is the data-source site or entry URL specified at deployment. It may be a match preview page, event page, or result-query website.
6. `bet_asset` is the betting asset name, using the SatoshiNet asset layer `AssetName.String()` format.
7. `min_bet_unit` is the minimum betting unit, expressed as a string and parsed as Decimal according to the asset properties of `bet_asset`.
8. `outcomes` is the candidate outcome list. The number of outcomes is variable, but at least 2 outcomes are required.
9. `outcomes.id` uses lowercase letter identifiers, for example `a`, `b`, `c`. IDs must be unique within the same contract.
10. Deploy payload does not include fee ratio, Agent policy, bootstrap address, or Agent address. These are protocol built-in configuration.

### Built-In Configuration

Prediction natural language contracts use protocol built-in configuration in the first phase:

1. Deployer fee: 6% of the prize pool.
2. Agent fee: 3% of the prize pool.
3. Bootstrap node fee: 1% of the prize pool.
4. Winner pool: 90% of the prize pool.

If a 3-Agent multi-Agent mode is adopted later, the total Agent fee remains 3%, and each Agent that participates in the valid confirmation result receives 1%. In the current single-Agent mode, CoreNode Agent receives the full 3%.

The bootstrap node receiving address is derived from the protocol built-in bootstrap public key. The current reference implementation can use `GetBootstrapAddress()` in `indexer/common/btc.go`. The Agent receiving address is derived from the protocol built-in Agent/CoreNode public key. A future implementation can add an Agent address derivation function corresponding to `GetBootstrapAddress()`.

### Betting Rules

After a prediction natural language contract enters `Ready` state, its internal state enters `Betting`, and users can participate through `bet` calls.

`bet` call parameters:

```json
{
  "outcome_id": "a"
}
```

Betting rules:

1. Any address can call `bet`.
2. The contract must be in the common `Ready` state, and the prediction contract internal state must be `Betting`.
3. Current time or block height must be less than or equal to `bet_deadline`.
4. `outcome_id` must exist in the deploy payload `outcomes` list.
5. The betting asset must be the deploy payload `bet_asset`.
6. The betting amount is determined by the Call TX funding output transferred to the contract address, not by any amount in call parameters.
7. The betting amount must be greater than or equal to `min_bet_unit`.
8. The betting amount must be an integer multiple of `min_bet_unit`.
9. The same address may bet on multiple candidate outcomes.
10. If the same address bets on the same outcome multiple times, state aggregates the amount by address and outcome. History keeps every invocation.

Each outcome bet must independently satisfy the minimum betting unit and integer-multiple rules.

### Result Confirmation

`confirm` is the Agent result-confirmation interface of prediction natural language contracts. It can only be called by CoreNode Agent. In the first phase, the default trigger is a `confirm` call.

`confirm` call parameters:

```json
{
  "result_type": "outcome",
  "outcome_id": "a",
  "source_url": "https://xxx.com",
  "result_url": "https://xxx.com/match/result/123",
  "result_hash": "...",
  "observed_at": 1780314000,
  "agent_version": "agent-v1",
  "model_version": "model-v1",
  "core_node_pubkey": "...",
  "core_node_signature": "..."
}
```

Confirmation rules:

1. The caller must be CoreNode Agent.
2. The contract must have entered `Ready` state.
3. Current time or block height must be greater than `bet_deadline`.
4. Current time or block height must have reached `confirm_after`.
5. `result_type` must be one of `outcome`, `cancelled`, `invalid`, or `unverifiable`.
6. When `result_type` is `outcome`, `outcome_id` must exist in the deploy payload `outcomes` list.
7. When `result_type` is `cancelled`, `invalid`, or `unverifiable`, `outcome_id` can be empty and the contract enters full-refund flow.
8. `source_url` must equal the `source_url` in deploy payload.
9. `result_url` is the concrete page URL actually used by the Agent to confirm the result. It may differ from the deploy payload `source_url`.
10. `result_url` must belong to the website scope specified by deploy payload `source_url`: the scheme must be the same, and the hostname must be the same hostname or one of its subdomains. If HTTP redirects occur, the final URL after redirects must also satisfy the same rule.
11. `result_hash` records the hash of the cleaned text used by the Agent when parsing the `result_url` page or data source.
12. `observed_at` records the time or height at which the Agent observed the result.
13. `agent_version` and `model_version` record the Agent implementation version and model version used to generate the confirmation.
14. `core_node_pubkey` must equal the protocol-configured CoreNode public key.
15. `core_node_signature` is the CoreNode signature over the confirmation payload. The current reference implementation accepts the node wallet message-signature format and keeps Schnorr signature verification compatibility.

The CoreNode signature payload must use the fixed domain:

```text
satoshinet.agent.prediction.confirm.v1
```

The signature covers domain, contract address, action=`confirm`, `result_type`, `outcome_id`, `source_url`, `result_url`, `result_hash`, `observed_at`, `agent_version`, and `model_version`. It does not cover `core_node_pubkey` or `core_node_signature` themselves.

Agent uses the deploy payload `source_url` to determine the allowed data-source scope and finds the concrete result page `result_url` within that website scope. Agent parses `result_url`, obtains the final event result, and submits a structured result through `confirm`. The contract runtime only handles structured results and does not directly rely on unstructured webpage content for fund distribution.

The cleaned text should remove scripts, styles, ads, navigation, and obvious dynamic noise, keeping only the main text used by the Agent to determine the result. In the first phase, consensus nodes verify the CoreNode signature and structured confirmation result, and do not re-fetch webpages in the block-validation path. `result_hash` is an on-chain notarization of the material used by CoreNode, available to indexers, explorers, audit tools, and future challenge mechanisms.

If multi-Agent mode is adopted, multiple Agents submit `confirm` results separately. The contract runtime confirms the final result when the same `result_type` and same `outcome_id` reach the protocol threshold, then automatically settles or refunds.

### Automatic Settlement and Refund

Prediction natural language contracts do not provide externally callable `settle` or `refund` interfaces. Normally, `confirm` triggers automatic settlement or refund.

When there are winners, the contract distributes all `bet_asset` funds in the contract address in the following order:

1. 6% to deployer.
2. 3% to Agent.
3. 1% to bootstrap node.
4. Remaining 90% to winners in proportion to their winning bet amount over total winning bet amount.

Fees and winner distribution use Decimal calculation. Remainders from ratio calculation enter the winner pool. Remainders from winner distribution are allocated by winning bet amount from high to low; for equal amounts, by address lexicographic order.

When there are no winners, the contract charges no fee and refunds all `bet_asset` funds in the contract address in proportion to original bet amounts.

The following cases enter full-refund flow with no fee:

1. match or event is cancelled.
2. the data source is accessible but the result is not in the candidate outcome list.
3. Agent confirms the result as unverifiable or invalid.
4. after `ready`, the contract content is found to be non-executable and settlement has not completed.

`confirm_after` is the expected time or height when the Agent starts checking the result. After `confirm_after` is reached, the Agent starts checking the data source and attempts to confirm the result. If the Agent has not confirmed yet, the contract does not refund automatically and continues waiting for Agent processing.

Once `bet_asset` funds enter the contract address, they are treated as part of the prediction contract prize pool and are not filtered by source. Assets other than `bet_asset` do not participate in the current prediction contract settlement.

### No Deployer Cancellation Interface

Prediction natural language contracts do not provide a deployer-only `cancel` interface.

Reasons:

1. Before the contract enters `Ready`, it cannot accept bets and usually has no funds that require deployer-triggered refund.
2. During `Betting`, allowing deployer cancellation creates risks of early termination, avoiding future results, or harming bettors.
3. During `ClosedForBet` or `PendingResult`, result-confirmation authority belongs to Agent. Deployer cannot replace Agent judgment.
4. When the match is cancelled, result is invalid, data is unverifiable, or the contract cannot execute, CoreNode Agent should submit `cancelled`, `invalid`, or `unverifiable` through `confirm` and trigger full refund.
5. If the Agent has not confirmed after `confirm_after` is reached, the protocol chooses to continue waiting for Agent processing and does not authorize deployer to bypass Agent for refund.

### Internal State

Prediction natural language contracts use the common `Ready` state to indicate that the contract has passed CoreNode review and can accept business calls. Prediction business maintains its own internal states:

1. `Betting`: users can bet.
2. `ClosedForBet`: betting is closed and no new bets are accepted.
3. `PendingResult`: waiting for Agent to submit the final result.
4. `Confirmed`: result has been confirmed by Agent or reached multi-Agent threshold.
5. `Settled`: prize distribution has completed and the contract is closed.
6. `Refundable`: the contract enters refund flow.

`Betting` is not a common contract state. It is only an internal state of prediction natural language contracts.


First-Phase Scope
----

Natural language contracts may first enter the SatoshiNet contract-result flow in single-CoreNode-Agent mode. After deployment, the contract must first be confirmed by CoreNode through the `ready` interface, and the confirmation result is written into chain state through canonical `CONTRACT_RESULT`.

The first phase still needs to define Agent execution details. Before full mainnet consensus execution, the protocol must define model determinism, verification method, challenge mechanism, arbitration mechanism, multi-Agent threshold rules, and state root commitment rules.
